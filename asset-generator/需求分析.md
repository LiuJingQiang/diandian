# 素材生成器 · 需求分析

## 1. 背景与目标

参照点点穿书（`com.cyclotron.in`）的视觉小说架构（见 `../命运币的代价/game/研究资料/点点游戏架构规范.md`），
把「已写好的小说」自动变成「带美术素材、可玩的互动读书游戏」。

作者的痛点：**小说有了，素材为零**。本工具解决「小说 ↔ 素材 的自动对应与生成」，
让作者不用手动切图、不用手动配置，一条命令补齐游戏所需的全部美术。

**核心原则：小说创作 与 素材生成 解耦。** 二者通过「素材需求规格(manifest)」这份契约衔接，
小说更新 → 重跑 → 只补新增/变动的素材。

## 2. 范围

| 属于 | 不属于 |
|---|---|
| 小说(散文+设定) → 故事图 JSON | 剧情分支创作（作者负责）|
| 自动列出游戏所需素材槽位 | 游戏引擎/前端（另见 game/）|
| 调 OpenRouter 文本模型补美术描述 | 真人约稿/素材站采买（人工可选路径）|
| 调 OpenRouter 图像模型出图 | 音频/视频生成（暂不做，仅留字段）|
| 产物直接落盘进游戏项目 | 游戏后端/账号/支付 |

## 3. 输入 / 输出（数据契约）

**输入（小说项目 `<project>/`）**
- `正文/第00X章.md`：章节散文
- `设定/核心设定.md` `关系.md`：角色/关系（供说话人归属与外貌起草）
- `book.config.json`：**小说专属配置**（书名/角色名单/变量/首章抉择/解锁成本）——换书只改它
- `设定/美术设定.json`：**素材生成输入**（全局画风 + 每角色外貌 + 每场景描述）——作者可填，缺则 LLM 起草

**输出**
- `game/story.<id>.json`：引擎故事图（节点=章节，边=选项/自动跳转）
- `game/books.json`：书架
- `game/assets/manifest.json`：**素材需求规格**（逻辑 key → 本地文件）
- `game/缺素材清单.md`：人读的待补清单 + 提示词
- `game/assets/{covers,characters,scenes}/*`：**生成的素材**（直接可用）

## 4. 功能需求

- **F1 转换**：散文按引号/【】切成 chats（旁白/对话，就近归属说话人）；章节串成图；章尾注入付费/广告选项（点点经济模型）。
- **F2 素材需求推导**：从故事自动枚举全部素材槽位（封面/每角色立绘+头像/每章场景），生成 manifest + 清单。
- **F3 美术设定脚手架**：生成/合并 `美术设定.json`；**合并时保留作者已填字段**，只补新增角色/场景。
- **F4 文本侧生成外移**：角色外貌、场景描述、BG 转场、choices 由同级 `../novel-generator/` 的 `text_model` 负责。
- **F5 出图(gen)**：按「全局画风 + 主体描述 + 画幅 + 负面词」拼提示词，调本项目 `image_model` 出图。
- **F6 直接落盘**：图片写入 manifest 指定目录并**回写 manifest** 指向真实文件；引擎刷新即生效。
- **F7 选择性生成**：`--only <key>` / `--kind <类>` / `--all` / `--force`(重生成) / `--dry-run`(只看提示词)。
- **F8 多项目**：`--project <路径>` 指向任意小说；缺省用 `gen_config.json` 的 `default_project`。

## 5. 非功能需求

- **N1 缺素材可跑**：任何素材缺失，游戏显示带标签占位图，不崩。
- **N2 零第三方依赖**：仅 Python 标准库（urllib），无需 pip install。
- **N3 密钥配置**：key 三处可配(优先级)：环境变量 `OPENROUTER_API_KEY` > `gen_config.local.json`(gitignore,推荐) > `gen_config.json` 的 `api_key` 字段。
- **N4 省钱可控**：`--dry-run` 免费预览提示词；建议先出单张验证再批量。
- **N5 一致性**：同角色多图靠 `美术设定.json` 的 `appearance` 固定特征，可选 `seed`/`refImage`。
- **N6 可换模型**：小说/文本模型只在 `../novel-generator/gen_config.json` 的 `text_model` 配置；图片模型只在本项目 `gen_config.json` 的 `image_model` 配置，且须支持图像输出。
- **N7 通用**：转换器不硬编码书名/角色，全部来自 `book.config.json`。

## 6. 关键设计决策

1. **逻辑 key 而非硬编码路径**：故事只引用 `char_shenmo` 这类 key，key→文件在 manifest。美术与剧情/代码彻底解耦。
2. **模型分工**：`novel-generator` 用文本模型把小说语言翻成 story/choices/BG/美术描述；`asset-generator` 只用图片模型出图。作者可在中间人工干预（改 `美术设定.json`）。
3. **合并而非覆盖**：重跑转换器不破坏作者填写的美术设定与已生成素材。
4. **多形态回包解析**：不同图像模型返回结构不同，按 `message.images → content[] → 正则兜底` 依次解析。

## 7. 边界与风险

| 风险 | 处理 |
|---|---|
| 说话人归属出错（旁白读了对话） | 就近归属 + 未知留旁白；可后续人工校正故事 JSON |
| 图像模型不支持图像输出 | 报「未解析到图片」并打印回包片段；换模型 |
| 角色跨图不一致 | 填 appearance/seed/refImage |
| 批量烧额度 | 先 `--only cover_book` 验证，再 `--all` |
| 换书忘改配置 | `book.config.json` 缺失时直接报错中止 |
